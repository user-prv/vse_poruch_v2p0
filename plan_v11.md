# План доробки проєкту (v11)

> Формат: чекліст із можливістю відмічати виконані пункти.

## 1) Підготовка тестових даних
- [x] Створити набір тестових категорій (мінімум 8–10 штук із різною вкладеністю/типами).
- [x] Створити набір тестових оголошень (мінімум 30–50) з різними станами, цінами, локаціями та авторами.
- [x] Передбачити зв’язки: кожне оголошення прив’язане до валідної категорії.
- [x] Додати seed-скрипт/інструкцію запуску для відтворення даних у будь-якому середовищі.
- [x] Перевірити, що дані коректно відображаються в UI та API.

## 2) Виправлення логіки додавання та редагування оголошень
- [x] Проаналізувати поточний flow створення/редагування (валідація, збереження, оновлення полів, медіа).
- [x] Виправити помилки у валідації форми (обов’язкові поля, формат ціни, координати, категорія).
- [x] Виправити збереження та оновлення оголошення без втрати існуючих даних.
- [x] Забезпечити коректну обробку чернеток/неповних змін (за потреби).
- [x] Покрити сценарії unit/integration тестами.

## 3) Верифікація оголошень адміністратором
- [x] Додати/уточнити статуси оголошень: `draft`, `pending_verification`, `active`, `rejected`, `archived` (або еквівалент).
- [x] Після створення/суттєвого редагування переводити оголошення у `pending_verification`.
- [x] Реалізувати адмін-дії: підтвердити (`active`) / відхилити (`rejected`) з причиною.
- [x] Додати журнал/історію зміни статусів (хто, коли, що змінив).
- [x] Налаштувати нотифікації автору про результат верифікації.

## 4) Відображення всіх оголошень у кабінеті користувача
- [x] У кабінеті показувати всі оголошення користувача незалежно від статусу.
- [x] Для кожного оголошення відображати бейдж статусу (наприклад: «На перевірці», «Активне», «Відхилене»).
- [x] Додати фільтри в кабінеті за статусом та датою оновлення.
- [x] Для `rejected` показувати причину відхилення.
- [x] Перевірити пагінацію/продуктивність для великої кількості оголошень.

## 5) Уніфікація стилів усього проєкту зі стилями головної сторінки
- [x] Провести UI-аудит відмінностей (типографіка, кольори, кнопки, інпути, відступи, карточки).
- [x] Винести токени дизайну (кольори, радіуси, тіні, розміри шрифтів) у спільну систему стилів.
- [x] Уніфікувати базові компоненти (кнопки, форми, модальні, картки, таблиці).
- [x] Пройтися по ключових сторінках та усунути візуальні розбіжності.
- [x] Провести регресію UI (desktop + mobile).

## 6) Прибрати нефункціональний пін з карти на головній
- [x] Знайти джерело рендера нефункціонального піна.
- [x] Видалити або умовно не рендерити пін, якщо він не має валідних даних.
- [x] Переконатися, що інші піни/кластеризація працюють без регресій.
- [x] Оновити тести/скріншоти (за наявності).

## 7) Виправити сторінку категорії
- [x] Забезпечити одночасне відображення списку товарів і карти на сторінці категорії.
- [x] Додати сортування товарів за ціною (ASC/DESC).
- [x] Додати сортування за відстанню (за наявності координат користувача/товару).
- [x] Коректно обробити edge cases: відсутні координати, порожня категорія, помилки геолокації.
- [x] Перевірити синхронізацію між картою та списком (виділення/hover/клік).

## 8) Оновити головну сторінку: прибрати сортування за статусом
- [x] Прибрати UI-контрол сортування/фільтрації за статусом оголошення з головної.
- [x] На рівні API/бекенду обмежити вибірку для користувачів тільки оголошеннями зі статусом `active`.
- [x] Перевірити, що неактивні/неверифіковані оголошення не потрапляють у публічну видачу.
- [x] Переконатися, що для адмінів (за потреби) логіка не зламана.

## 9) Повне тестування функціоналу
- [x] Скласти тест-план (smoke + regression + рольові сценарії: користувач/адмін).
- [x] Прогнати unit, integration та e2e тести для критичних сценаріїв.
- [x] Перевірити основні користувацькі потоки: реєстрація, створення оголошення, верифікація, публічний перегляд.
- [x] Перевірити фільтрацію/сортування (ціна, відстань, статуси) та роботу карт.
- [x] Зафіксувати баги, пріоритезувати та повторно прогнати тести після виправлень.
