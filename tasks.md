# План міграції `vseporuch-old` на стек Golang + React

## 0) Підготовка репозиторію
- [ ] Перенести/підключити старий PHP-код у папку `vseporuch-old/` (якщо ще не додано).
- [ ] Зафіксувати цільову структуру монорепозиторію:
  - [ ] `backend/` — API на Go (Gin + GORM + Logrus).
  - [ ] `frontend/` — клієнт на React.
  - [ ] `vseporuch-old/` — legacy PHP як референс на перехідний період.
- [ ] Додати `README.md` з коротким описом архітектури та сценарію запуску кожної частини.

## 1) Аудит legacy-системи (PHP)
- [ ] Проінвентаризувати модулі, ендпоінти, cron/jobs, інтеграції та ролі користувачів.
- [ ] Скласти карту бізнес-функцій: "що є", "що критично", "що можна відкласти".
- [ ] Зафіксувати поточну схему БД, зв’язки, індекси, обмеження та міграції.
- [ ] Визначити технічний борг і ризики під час міграції (безпека, продуктивність, сумісність).

## 2) Проєктування цільової архітектури
- [ ] Спроєктувати API-контракти (REST/JSON) для всіх критичних сценаріїв.
- [ ] Визначити шари `backend/`: `handler -> service -> repository -> db`.
- [ ] Визначити підхід до логування (Logrus): формат, рівні, correlation/request id.
- [ ] Визначити стратегію доступу до віддаленої БД (env-конфіг, SSL/TLS, пул з’єднань, таймаути).
- [ ] Затвердити модель стану на фронтенді (React state management, API client, error handling).

## 3) Ініціалізація `backend/` (Golang)
- [ ] Створити Go-проєкт у `backend/` з базовими залежностями: `gin`, `gorm`, `logrus`.
- [ ] Реалізувати конфігурацію через env (`.env.example`): `DB_HOST`, `DB_PORT`, `DB_NAME`, `DB_USER`, `DB_PASSWORD`, `DB_SSLMODE`.
- [ ] Налаштувати підключення до віддаленої БД через GORM + healthcheck ендпоінт.
- [ ] Додати middleware у Gin: recovery, request logging, CORS, request id.
- [ ] Впровадити базову структуру модулів (auth/users/... відповідно до legacy-аудиту).

## 4) Ініціалізація `frontend/` (React)
- [ ] Створити React-проєкт у `frontend/` (Vite або інший узгоджений інструмент).
- [ ] Налаштувати структуру: `pages/`, `components/`, `services/api/`, `hooks/`.
- [ ] Реалізувати базовий API-клієнт (base URL, обробка помилок, retry/pending-state).
- [ ] Підготувати базові layout/route guards для ролей та авторизації.
- [ ] Додати змінні оточення (`.env.example`) для URL бекенду.

## 5) Поетапна міграція функціоналу
- [ ] Пріоритезувати фічі у хвилі міграції (Wave 1/2/3).
- [ ] Для кожної фічі виконувати цикл:
  - [ ] Описати API-контракт.
  - [ ] Реалізувати backend endpoint + service + repository.
  - [ ] Реалізувати відповідний UI у React.
  - [ ] Провести функціональну перевірку проти поведінки `vseporuch-old`.
- [ ] Вести migration checklist відповідності "legacy -> new".

## 6) Дані та сумісність
- [ ] Підготувати SQL/GORM-міграції (якщо потрібна еволюція схеми).
- [ ] Реалізувати скрипти міграції/синхронізації довідників та критичних даних.
- [ ] Перевірити зворотну сумісність форматів даних для фронтенду та інтеграцій.
- [ ] Описати rollback-процедури у випадку невдалого релізу.

## 7) Якість, безпека, спостережуваність
- [ ] Додати базові unit/integration тести для ключових backend-сервісів.
- [ ] Додати frontend-тести для критичних користувацьких сценаріїв.
- [ ] Перевірити authN/authZ, валідацію вхідних даних, захист секретів.
- [ ] Налаштувати централізовані логи (Logrus), метрики та алерти.

## 8) Реліз і decommission legacy
- [ ] Запустити staging з повним smoke/regression тестом.
- [ ] Провести канарний або поетапний прод-реліз.
- [ ] Моніторити помилки/латентність і оперативно закривати інциденти.
- [ ] Після стабілізації — формально відключити `vseporuch-old` і оновити документацію.

## Definition of Done
- [ ] Усі критичні legacy-сценарії працюють у новому стеку (Go + React).
- [ ] `backend/` і `frontend/` мають інструкції запуску та деплою.
- [ ] Віддалене підключення до БД стабільне, логування через Logrus активне.
- [ ] Legacy PHP не використовується у production-трафіку.
